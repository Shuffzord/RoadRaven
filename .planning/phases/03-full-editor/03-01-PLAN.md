<objective>
Phase 3 foundation. Build the Zustand mutation action family, the single canvas-level keyboard router, the floating inline rename input, the confirmation dialog for non-leaf deletes, the clipboard subtree serializer, and the dashed focus ring that coexists with the solid selection ring.

Purpose: every other Phase 3 plan depends on these store actions. Context menu (Plan 02) calls them, Side panel editor (Plan 03) calls them, autosave (Plan 04) subscribes to them. Doing them first in strict dataKey discipline is the only way to hit the 30 fps gate while shipping EDIT-01..EDIT-08.

Output:
- Store mutation actions with the structural / in-place discipline from RESEARCH.md Pattern 1
- Keyboard router that routes shortcuts context-aware via `document.activeElement`
- Floating `<input>` overlay with inverse D3 zoom transform math (D-13, EDIT-01)
- ConfirmationDialog component for non-leaf delete (EDIT-03)
- Clipboard subtree serializer with read-permission fallback (A2)
- Focus ring (dashed) distinct from selection ring (solid), coexistence rules per D-06
- Full unit-test coverage for every mutation
</objective>

<design_note>
**Context menu NOT handled in this plan.** D-02's "custom webview-rendered `<div>`" is honored by Plan 02 using Radix Context Menu (a headless library that renders a custom `<div>` via portal — it is NOT a native OS menu). This plan only provides the store actions the menu will call plus the keyboard router that is the parallel entry point to the same actions.

**Focus / selection model (D-06):** two Zustand fields — `focusedNodeId` (arrow-key nav) and `selectedNodeId` (panel). Space copies focus → selection. Arrow keys mutate focus only. Both rings may coexist on the same node; CSS uses `outline` for selection, `box-shadow` for focus ring to avoid the CSS outline stacking limitation.

**dataKey discipline (research Pattern 1):**
- BUMP dataKey: addChild, addSiblingAbove, addSiblingBelow, deleteNode, duplicateNode, moveNodeUp/Down, renameNode, pasteNode
- DO NOT bump dataKey: updateNodeStatus (already shipped), updateNodeType, updateNodeMetadata, updateNodeNotes, setFocusedNode, setSelectedNode
- `statusTick` already exists for in-place counter — reuse for type/metadata/notes changes

**Rename bumps dataKey.** Per RESEARCH.md Pattern 1, renaming changes `nodeDatum.name` which Canvas reads from treeData. Cleaner to rebuild treeData + bump dataKey on rename than to refactor renderCustomNodeElement. Rename frequency is low enough that the full re-render is acceptable.

**Clipboard fallback (Pitfall 6, Assumption A2):** Ctrl+C always writes to `navigator.clipboard.writeText` AND stores a copy in a module-level `lastCopiedSubtree` buffer. Ctrl+V tries `readText` first; on `NotAllowedError` or other denial, falls back to the buffer. Cross-app paste may fail silently — acceptable per research.
</design_note>
