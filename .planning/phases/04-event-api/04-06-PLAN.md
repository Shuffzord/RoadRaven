### UAT-1 — Pulse animation invisible (Test 1, severity: major)
> "I dont see any pulse animation nor any visible changes in the node or app itself. The only thing updating is the status, metadata and event log."

**Root cause (CORRECTED — replaces the previous theory):** The previous root-cause analysis claimed the inline `style={{ boxShadow: "var(--rv-shadow-node)" }}` on `RoadmapNode.tsx:120` shadowed the `@keyframes rv-node-pulse` animation. **That theory was wrong.** Per the CSS Animations spec (https://www.w3.org/TR/css-animations-1/#animations), an active animation overrides author rules INCLUDING inline styles for the duration the animation is running on the animated property. The inline `boxShadow` does NOT block the keyframe — that fix would have been a no-op.

The actual root cause, confirmed by reading `Canvas.tsx:266-270`, is **foreignObject clipping**. Every node card is rendered inside:

```jsx
<foreignObject width={240} height={100} x={-120} y={-50} overflow="visible">
  <RoadmapNodeCard ... />
</foreignObject>
```

The card lays out at (0,0) inside its 240×100 foreignObject. The current pulse keyframe paints `box-shadow: 0 0 0 6px var(--rv-status-completed-bg)` — a 6px outset ring extending into the foreignObject's negative coordinate space (e.g. x=-6, y=-6 on the top-left edge). Although `overflow="visible"` is set as an SVG attribute, Chromium/CEF foreignObject has long-standing clipping bugs where outset `box-shadow` from descendants is still cropped at the foreignObject's content-box boundary. The user sees zero visual change → UAT-1 reproduces.

**The fix (option 3 — chosen by user):** Render the pulse as a `.node::after` pseudo-element with `inset: -3px`, painted *inside* the card's stacking context (the card div already has Tailwind `relative` on `RoadmapNode.tsx:112`, establishing the positioning context). The animated property is `border-width` on the pseudo-element (0px → 3px → 0px), not outset `box-shadow` on the card itself. Because the pseudo-element is laid out as a child of the card div and its border paints within its own bounding rect, the painted ring stays inside the card's stacking context. The 3px outward extension via `inset: -3px` is well within the foreignObject's headroom (10px horizontal, 22px vertical clearance for a ~220×55 card in a 240×100 foreignObject), so even residual foreignObject clipping cannot crop the ring at its strongest 3px width.

The inline `boxShadow: "var(--rv-shadow-node)"` on `RoadmapNode.tsx:120` STAYS — it was never the bug, it provides the static drop-shadow that all node cards (live or not) need. The `::after` ring composes visually on top.
