### UAT-1 — Pulse animation invisible (Test 1, severity: major)
> "I dont see any pulse animation nor any visible changes in the node or app itself. The only thing updating is the status, metadata and event log."

**Root cause hypothesis (confirmed by code read):** `RoadmapNode.tsx` sets `style={{ boxShadow: "var(--rv-shadow-node)" }}` inline on the `.node` element (line 112). The pulse animation defined in `index.css` (lines 724–744) animates the same `box-shadow` property. Inline styles win against keyframe animations on the same property without `!important` (CSS spec — animations rank between author rules and `!important` only when no inline style targets the animated property; an inline `box-shadow` declaration sits at the same specificity tier and overrides the animation values). Result: `data-live="true"` is set in the DOM, the keyframe is defined, but no visual change happens because the inline `boxShadow` rules every frame.

The fix moves the static shadow off `box-shadow` and onto a CSS custom property (`--node-shadow`) that the keyframe composes into its `box-shadow` declaration. The animated outer ring is added as a second shadow layer in the keyframe — no inline override possible.

### UAT-3 — Connected pill click does not open drawer (Test 3, severity: major)
> "Saw `● :47921 · 1` (connected, 1 producer) — clicking the pill did not open the event log drawer. Idle URL-copy works."

**Root cause hypothesis (confirmed by code read):** `EventApiPill.tsx` lines 9–22 use a `@vite-ignore` runtime-constructed dynamic import for `eventLogStore` to defeat Vite static analysis. This was added when `eventLogStore` did not yet exist (Plan 04-04 was downstream). The store has shipped; the dynamic-import workaround now silently fails in some bundle modes (verification report flagged this as Info-level anti-pattern). The fix replaces the dynamic import with a plain `import { useEventLogStore } from "../store/eventLogStore"` and calls `useEventLogStore.getState().setOpen(true)` directly in the connected branch.

### UAT drive-by — Drawer cannot be closed (severity: major)
> "Drive-by observation during Test 3 setup: drawer opened via the node integration panel `Open full log →` button cannot be closed by the user. Likely missing close affordance (X button, ESC handler, or click-outside) or the close action is wired to a setter that no longer fires."

**Root cause:** `Ctrl+Shift+L` toggles the drawer (verification truth #22), but no visible affordance exists to close it. There is no `[×]` button, no Escape handler, and no other discoverable mechanism. Users open the drawer via `IntegrationZone`'s "Open full log →" link without learning the keyboard shortcut.

The fix adds (a) a `[×]` close button in the drawer header strip area with `aria-label="Close event log"`, and (b) an Escape handler in `useKeyboardRouter.ts` that closes the drawer **only** when focus is inside the drawer region — preserving Phase 3's existing Escape-cancels-rename / Escape-deselects-node semantics. Click-outside is intentionally NOT added (drawers are non-modal — clicking the canvas should not close the drawer; mirrors VS Code panels).
